Take a moment and review the two repos referenced in claude.md. I want to cherry pick concepts from them for this new project, which is a distributed ai system across various mac minis, pcs, mobile phones, raspberry pis and occassionally cloud services. the goal is to create a distribuated system that serves as the personal ai assisant for my family while keeping as much of our personal information OUT of cloud providers. I don't need OpenAI or Anthropic training their data on our families personal data. I explored various concepts in the two other project repos, but i short I want to create:

An AI assistant that runs on a device I wholly own (at first this will be a mac studio in my garage). The mac studio has a lot of ram and cpu and will carry out ai intensive projects, but we may move the database for the solution to an accompanying mac mini to make sure it doesn't get overwhelmed / begin running slowly.
It should support the concept of "events" that are propogated across all connected listening device. For example, there may be a raspberry pi that has a web portal instance open that visualizes an ai avatar. that device would have a camera attached and running facial recognition. when it notices me or my wife walk by, it should surface an event back to the server "saw tim walk by" and any system that cares can then consume that event. that is the most simple version. it might one day scale up to an autonomous robot surveying a farm field and surfacing lots of different events - but that is a long term goal. 
It should have consistently running background processes, perhaps run through prefect or something like that, which will help in data analysis, coding projects, research, etc. 
it should support the concepts of "clients" to the system, and clients may/will come and go over time - with new ones being connected and old ones being removed as necessary. 
In the mia-sempre project there is a document that talks about a brain to nerve hierarchy system. that is a neat concept for the autonomous robots and our event architecture should attempt to be ready for it, but not ncessary implement it day one. 
The main way we will interact with this system is through "client" instances such as a custom made mobile app, web portals (either on laptops or in static installations throughout the home / office). clients obviously should not be able to just immediately connect. they should register themselves, and then be granted access into the system. Also, clients should have a concept of priveledge. A client registered to my son should not have access to data about me or my wife, but my wife and I should be able to query the system about our sons data. Static installations should have cameras installed and when they detect a person they can talk to that person with limited ability to access their data. In short, I wouldn't want a static installation to see me and be like "hey tim, how is your rash" when there may be other people within earshot. 
as mentioned, i want to build a mobile app and a web portal. both will support access to my family members only: laurie (wife), bennett (son), drew (son), david (son), william (son). possibly some day ill expand it to other family members, but likely ill just let them install their own instance on their own servers rather than making a single instance support multiple families.
In the early days I'll likely need to rely on cloud ai services to accelerate the project, but long term I want the system to be able to wholly run locally using ollama or some other local system pattern. 
The system should support the idea of a long term project with a set of apis that govern project management. I had a concept like this in local-vida, but I want to be able to toss work to the system "jarvis-style" and put ideas on the backlog.
