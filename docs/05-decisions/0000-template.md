**Status:** template (copy and rename for new ADRs)

## Context

What's the situation that requires a decision? What constraints, prior decisions, or events motivate writing this ADR now? Keep this section factual — describe the problem, not the solution.

## Options

Lay out the options under serious consideration, each with pros and cons. Aim for at least two; if there's truly one, the ADR is documentation, not a decision.

## Decision

The chosen option, stated unambiguously. If still proposed/open, say so and why no decision yet.

## Consequences

What changes as a result of this decision? What becomes easy, what becomes hard? Where does the rest of the system need to adapt?

## References

- Related ADRs (supersedes / superseded by / depends on).
- Architecture docs that consume this decision.
- External material (RFCs, vendor docs, blog posts that informed the decision).

---

## Status values

- **proposed** — recommendation made, awaiting confirmation. Use the 🟣 marker.
- **open** — options laid out, no decision yet. Use 🔵.
- **accepted** — decided, can be relied on. Use 🟢.
- **superseded** — replaced by a later ADR. Link to the successor.
